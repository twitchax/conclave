# Summary

The local half of the fabric: a per-session MCP server that injects inbound channel/whisper traffic
into Claude Code and exposes the outbound tools, while enforcing the local permission policy by
capability. It connects out to one or more central servers and survives reconnects.

# Problem

Without the bridge there is no way for messages to enter or leave a Claude Code session. This
milestone also proves the load-bearing `claude/channel` capability against the current CC version
(the de-risk we folded out of the old M0 trial).

# Goals

1. An MCP stdio server that injects inbound messages as `<channel>`/`<whisper>` tags.
2. A multi-server WS client that reconnects and re-subscribes.
3. Permission enforcement by capability (mute/notify/converse/act) with call-time per-channel checks.
4. A pluggable notification sink (session pane in v1) and the carried-over permission relay.

# Technical Approach

The bridge is a dual peer (DESIGN §13): a stdio MCP server toward Claude Code and a
`tokio-tungstenite` WS client toward central. Inbound frames are resolved against the local
`(server, channel)` permission level; `mute` drops, otherwise the message is injected through a
`NotificationSink` with the level-appropriate surrounding prompt. Outbound tool calls are gated —
emit tools are present only when some joined channel is `>= converse`, and a call whose target
channel resolves lower is rejected at call time. A typestate machine models the connection
lifecycle. The e2e suite spawns real `serve` + `bridge` binaries.

# Assumptions

- The M2 server is complete (auth, channels, presence, fan-out).
- The experimental `claude/channel` capability exists in the installed CC (validated in T-001).

# Constraints

- Per-channel gating cannot be tool-absence (MCP tool lists are session-global) — it must be
  call-time rejection (DESIGN §9).
- CC ↔ bridge stdio is plaintext (local pipe).
