# Summary

Build the shared substrate the server and bridge both depend on: the versioned, E2E-ready wire
protocol; the machine identity/keystore (Ed25519 + `secrecy` + local config); and the embedded
SurrealDB schema behind a thin repository.

# Problem

The server (M2) and bridge (M3) can't be built until their common contract — frames, identity,
and durable storage — exists. Getting the wire format's version field and encrypted-payload
envelope right *now* is what keeps v2 E2E additive rather than a breaking change.

# Goals

1. A versioned frame schema with an opaque encrypted-payload envelope reserved for E2E.
2. Machine identity: Ed25519 keygen/sign/verify, `secrecy`-wrapped keys, local `config.toml`.
3. Embedded SurrealDB schema + typed repository with the uniqueness constraints from DESIGN §15.

# Technical Approach

Mirror ratrod's `protocol`/`base`/`security` split. Frames are an enum serialized with a fixed
bincode config; the `Hello` handshake negotiates a protocol-version integer (server rejects
incompatible peers). The data frame carries `{ payload | ciphertext-envelope, key_id }` so E2E
slots in later. Identity lives in `identity`, storing an Ed25519 key (private key `SecretBox`)
and a TOML config. The store is embedded SurrealDB accessed through a hand-written repository of
serde structs — the SDK is the typed layer, no ORM.

# Assumptions

- The M0 scaffold (lints, cargo-make, test harness) is green.
- A single embedded SurrealDB instance suffices (no HA in v1).

# Constraints

- The wire format must be forward-compatible from this PRD onward; later PRDs may add frame
  variants but must not renumber/repurpose existing ones without a version bump.
- Private key material must never be logged or written unencrypted to logs (may be plaintext at
  rest per DESIGN §12).

# Non-Goals (MVP)

- E2E encryption itself (only the envelope is reserved) — DESIGN §19.
- Any networking/serving (M2) or MCP/bridge (M3) behavior.
