# Summary

Make the bridge and CLI reliable and safe under real use: every deferred tool call terminates (a
result or an error, never a hang), responses are correlated explicitly instead of by arrival order,
inbound untrusted text cannot escape its injection framing, and CLI verbs never hang or silently
misconfigure. These are the defects an operator will *feel* the moment they drive a live server from
a session — including during PRD-0009 deployment testing.

# Problem

The post-v1 review confirmed two HIGH bridge defects that break response integrity: an out-of-band
`Error` frame pops the wrong entry off the per-server FIFO used to correlate MCP responses
(`bridge.rs:553`), permanently desyncing every subsequent tool answer; and pending request ids are
never cleared on a link drop (`bridge.rs:143`), so deferred tool calls **hang forever** after a
disconnect. Around these sit silent connect-failure handling, premature join confirmation,
unbounded CLI hangs against a dead server, non-atomic `config.toml` writes that can brick the CLI,
and an unescaped injection path that lets any sender forge a "trusted" block inside an agent
session.

# Goals

1. Explicit, collision-proof request↔response correlation; out-of-band frames never steal a call's answer.
2. Link drops fail pending calls promptly; no permanent hangs anywhere in the tool path.
3. Untrusted inbound bodies cannot break out of `<channel>`/`<whisper>` framing.
4. CLI verbs are bounded (timeouts), configuration writes are crash-safe, and scopes are explicit.

# Technical Approach

Replace order-based correlation with an explicit key: tag each deferred control request with a
client sequence number the server echoes on its reply (append-only protocol field), and resolve the
`pending` map by that key — Error frames from fire-and-forget sends route to the notice path and
never pop a deferred slot. Model link state as an internal event on the inbound channel so the
dispatcher can fail-and-clear all pending calls for a server before resubscribe. Wrap the control
client in `tokio::time::timeout`. Escape the injection delimiters in the sink so a body cannot emit
its own closing sentinel. Make `config.toml` writes temp-file-plus-rename under an advisory lock,
and require an explicit `--channel`/`--whisper`/default scope from `perm set`.

# Assumptions

- M0–M5 complete and green; server-side hardening (PRD-0007) is independent and can land in parallel.
- Any protocol addition (a correlation id) stays append-only and forward-compatible.

# Constraints

- Keep the hand-rolled JSON-RPC MCP peer and the transport-agnostic BridgeCore design (no framework swap).
- Permission semantics (mute/notify/converse/act, whisper as a distinct scope) are unchanged — only their plumbing is hardened.

## 2026-07-03 — T-001/T-002: link-driven response correlation

The bridge's per-server FIFO (`pending`) could not correlate correctly: `send_channel`/`whisper`
replied "sent" immediately and the server acked nothing on success, so their *failure* `Error`s were
orphans that popped an unrelated deferred call's slot (#3); and a link drop never cleared `pending`,
so deferred calls hung forever (#4). Reconnect resubscribe `Join`s (issued in `client.rs`, bypassing
the dispatcher) drew `Joined` acks that likewise stole slots.

Fix (chosen with the user — confirmed sends + link-driven pending):
- **Server acks successful `ChannelMsg`/`Whisper`** (`session.rs`; the `Ack` variant already exists —
  no wire change). Sends are now *confirmed*: `send_channel`/`whisper` defer and their result
  reflects real delivery, and their errors correlate instead of misrouting.
- **`run_link` emits `LinkEvent::{Up,Down}`** on the shared inbound channel; resubscribe moved out of
  `client.rs` into the dispatcher. On **Down** the dispatcher fails every pending tool call for that
  server ("connection lost; retry") and clears the queue (#4). On **Up** it re-subscribes each joined
  channel, enqueuing a silent `Resubscribe` sentinel per `Join` so its `Joined` ack is consumed
  instead of answering a tool (#3). `pending` is now `VecDeque<Pending::{Tool,Resubscribe}>`.
- Every server-bound request now enqueues exactly one entry and the server responds to each in order,
  so the in-order FIFO is correct.

TDD: `test(/correlation/)` (a whisper-then-who where the whisper's Error and the who's Presence must
resolve their own ids, not swap), `test(/link_down_fails_pending/)` (a dropped link fails the pending
who), and a resubscribe-ack-is-silent test. Full gate green (145 tests). Note: sends are now
synchronous (one round-trip) rather than fire-and-forget — a UX improvement (real delivery
confirmation). The e2e server-spawn timing flake in `wait_for_listener` is pre-existing (retry-covered).
