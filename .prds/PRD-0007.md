# Summary

Close the security and robustness gaps that matter the instant `conclave serve` is reachable by
anyone but its operator. The centerpiece is identity-namespace integrity — no username or public
key is durably enrolled without proving possession — plus atomic membership mutations, elimination
of wire-reachable panics, discovery-leak closure, and pre-auth/backpressure DoS guards.

# Problem

v1 was built and proven on the happy path (M0–M5, `v0.1.0`). A post-v1 adversarial review
(42 agents, 28 confirmed findings) surfaced a cluster of network-facing defects that are latent
today only because the server has never been exposed. The most serious lets an **unauthenticated**
client durably squat usernames and enroll public keys it never proved it holds (`session.rs:132`
persists before the Auth proof at `:145`). Others allow concurrent single-use-invite double-redeem,
lost ACL entries, a reachable panic from a crafted `--expires-in`, private-channel existence
oracles, and unbounded pre-auth memory growth. All must be fixed before the deployment milestone
(PRD-0009) exposes the origin.

# Goals

1. No durable identity write without a verified possession proof (register and machine-add).
2. Admin allowlist bound to keys, not squattable names.
3. Atomic channel-record mutations: no lost ACLs, no invite over-redemption.
4. Referential integrity for invites/ACLs across channel delete/rename and user removal.
5. Zero network-reachable panics; uniform discovery errors; pre-auth timeout + frame-size + backpressure caps.

# Technical Approach

Reorder the handshake so `hub.register` runs only after `identity::verify` succeeds (or wrap
register+auth in a rollback path). Move channel-record read-modify-write behind a serialization
point — either an async `Mutex` held across the store round-trip, or (preferred, and the reason the
embedded SurrealDB SDK was chosen) a server-side conditional `UPDATE ... WHERE ... RETURN` inside a
transaction so redemption and ACL append are atomic. Add `Store` methods for invite cascade and
ACL/creator purge. Replace overflowing arithmetic with checked variants surfaced as
`ProtocolError`. Collapse `ChannelNotFound`/`NotMember` into one non-differentiating error for
non-members and re-gate `who()` on subscription. Enforce a handshake deadline (a pre-auth reaper
bucket), set tungstenite's max frame/message size to `Constant::MAX_FRAME_SIZE`, and bound the
per-session outbound channel with an explicit slow-consumer policy.

```
BEFORE:  Hello -> Challenge -> Register(persist!) -> [Auth verify]   <-- durable write before proof
AFTER:   Hello -> Challenge -> Register(stage) -> Auth verify -> commit | drop  <-- proof gates the write
```

# Assumptions

- M0–M5 are complete and green (`v0.1.0`); this is additive hardening on the existing server.
- TLS/edge termination and deployment are out of scope here (PRD-0009); fixes are transport-agnostic.

- Protocol changes, if any, remain append-only and forward-compatible (protocol version discipline).
- In-memory presence/subscription model is retained; fixes must not require durable presence.

## 2026-07-02 — T-001 (register path): verify-first handshake

The registration handshake now proves possession **before** any durable write. `session.rs`
previously called `hub.register` (durable `create_user` + `create_machine`) the instant a `Register`
frame arrived, then read and verified the `Auth` — so a bad signature, a mismatched key, or a
disconnect left the username squatted and the unproven key enrolled (finding #1, HIGH). The Auth is
now read and verified (`auth_pubkey == pubkey` + `identity::verify`) first; `hub.register` runs only
on success. Verify-first, so no rollback path is needed.

TDD: two `test(/register_proof/)` cases (bad signature, mismatched key) fail on the old ordering
with `username "aaron" is already taken` and pass after the reorder; full gate green (123 tests).
The disconnect-after-Register path is fixed by the same reorder but is not separately tested — a
deterministic version would race the server's frame processing (avoided per the no-sleep-races
convention). uat-001 reworded to match what is deterministically proven.

T-001 is `done`: register proves possession before any write, and `MachineAdd` is scoped to self-service enrollment with possession proven at first-connect auth (finding #8, documented + tested).

## 2026-07-02 — T-002: key-bound admin allowlist

The server-admin allowlist changed from `HashSet<String>` to `AdminAllowlist`
(`username -> Option<pubkey-b64>`). `serve --admin aaron=<pubkey>` pins the name to a key;
`Hub::register` now rejects any attempt to claim a pinned admin username with a non-matching key
(new typed `AuthError::Reserved`, mapping to `Unauthorized`) — closing the fresh-deploy admin-squat
(finding #7, HIGH). The pin also protects re-claims after `UserRemove`. Admin stays a per-username
property (`is_admin` is a key lookup), so every machine the bound user enrolls inherits admin, as
before. Bare `--admin aaron` still works but logs a squat warning at startup.

TDD: `test(/admin_bootstrap/)` — a non-matching key claiming a pinned admin name is rejected while
the bound key claims it and gets admin rights; it fails on the pre-enforcement build (attacker gets
`Established`) and passes after. Full gate green (124 tests).

## 2026-07-02 — T-003: atomic membership + invite redemption (DB-native)

The ACL double-spend/lost-update bugs (finding #2) are fixed **without an application mutex**, at
the DB layer, after empirically establishing that the embedded SurrealKV backend does **not**
reliably serialize concurrent writes to the *same* row — a 12-way concurrent `array::union` on one
channel row silently dropped an entry (no conflict error to retry on). Confirmed via web + the
pinned crate source: SurrealKV is optimistic-concurrency (`TransactionWriteConflict` →
`TransactionConflict`), and its own docs say the app must retry.

Fix, chosen with the user (option A — normalize):
- **Membership** moved from `ChannelRecord.acl: Vec<String>` to a `membership` table with a unique
  `(channel, user)` index. Concurrent joins `INSERT` **distinct keys** → no shared-row contention,
  no lost updates. Add is idempotent (`ON DUPLICATE KEY UPDATE`); membership cascades on channel
  delete and rename. `join`/`list_channels`/`who` now consult membership rows.
- **Invite uses** stay a single counter but use a **guarded atomic decrement**
  (`... SET uses_remaining = uses_remaining - 1 WHERE uses_remaining > 0 RETURN VALUE`): only one
  writer ever mutates, so concurrent redeemers of the last use are mutually exclusive. This path was
  reliable in stress testing (the guard means near-zero write contention).
- A bounded **retry-on-`conflict`** helper in the store covers any residual OCC conflict.

TDD: two concurrency tests — a 16-way single-use redeem admits exactly one; a 12-way concurrent
join preserves every membership. Both were flaky/failing before (the ACL one silently lost an
entry), and are now **30/30 clean with retries off**. Full gate green (127 tests). The ban/join
TOCTOU (#30) remains open.

## 2026-07-02 — T-004: referential integrity on delete/rename/user-removal

`delete_channel` now also drops the channel's invites (memberships were already cascaded in T-003),
and `rename_channel` migrates both invites and memberships to the new name — so a deleted channel's
outstanding tokens can no longer open a future same-named channel (#5). `UserRemove` now deletes
every channel the removed user created (cascading those channels' memberships + invites) and purges
the user's memberships elsewhere before deleting the account, so re-registering the freed username
inherits neither channel-admin rights nor private access (#6).

TDD: `test(/invites_/)` (invite dropped on delete, invite follows rename) + `test(/user_remove_purge/)`
(removed user's created channel deleted, membership elsewhere purged). Full gate green (130 tests).
